# Feature Specification: 客户端连接管理（建立连接阶段）

**Feature Branch**: `001-manage-client-connection`  
**Created**: 2025-12-30  
**Status**: Draft  
**Input**: 用户描述: "基于目前的进度 rtmate-server ClientConnection ，开发发起连接时对连接进行管理"

## User Scenarios & Testing *(mandatory)*

### User Story 1 - 连接建立时创建并跟踪客户端连接 (Priority: P1)

当业务方的客户端应用第一次通过 WebSocket 发起连接时，系统会为该连接创建一条“客户端连接记录”，
分配唯一连接标识，并在其整个生命周期内跟踪状态（例如：连接中、认证中、可用、已关闭）。

**Why this priority**: 如果没有清晰、可靠的连接记录，就无法在后续实现认证、订阅、广播、限流等能力，
这是实时内核最基础的能力之一，必须优先实现。

**Independent Test**: 仅启动最小实时内核，接入一个测试客户端建立连接，不进行任何业务消息收发，
验证是否自动创建连接记录、分配唯一连接 ID，并在连接关闭后将状态更新为“已关闭”。

**Acceptance Scenarios**:

1. **Given** 系统处于运行状态且没有现有连接， **When** 一个客户端成功完成 WebSocket 连接建立，
   **Then** 系统为该连接创建一条连接记录，并分配唯一连接 ID。
2. **Given** 一个已有连接处于可用状态， **When** 客户端主动关闭连接或网络异常断开，
   **Then** 系统将对应连接记录标记为已关闭，并记录关闭原因和关闭时间。

---

### User Story 2 - 基于连接记录维护认证与上下文 (Priority: P2)

当客户端建立连接并完成认证后，系统会在连接记录中绑定必要的身份 / 应用 / 租户等上下文信息，
以便后续所有消息处理都能基于这一上下文进行授权与隔离。

**Why this priority**: RTMate 面向多应用 / 多租户场景，连接如果没有绑定身份与上下文，
后续的频道、Presence、广播都会面临越权和混淆风险。

**Independent Test**: 在只实现“连接 + 简单认证”的场景下，通过一个测试客户端完成认证，
验证连接记录中是否包含应用标识、用户身份以及其他上下文字段；在认证失败时不写入这些信息。

**Acceptance Scenarios**:

1. **Given** 一个新建连接处于“待认证”状态， **When** 客户端发送合法的认证消息并通过验证，
   **Then** 系统在连接记录中绑定应用标识和身份信息，并将连接状态更新为“可用”。
2. **Given** 一个新建连接处于“待认证”状态， **When** 客户端发送无效或过期的认证信息，
   **Then** 系统不会在连接记录中绑定任何身份信息，并根据策略返回错误并关闭连接。

---

### User Story 3 - 管理异常与重复连接 (Priority: P3)

当同一业务实体（例如同一应用实例或同一用户）发起过多连接，或连接长期不活跃时，
系统能够基于连接记录进行控制，例如限制每个应用的最大并发连接数、关闭长时间不活跃的连接，
以保护内核资源并避免影响其他租户。

**Why this priority**: 在多租户的实时服务中，不受控的连接增长会迅速消耗资源，
导致性能问题甚至影响整体服务稳定性，需要基于连接管理能力做基础保护。

**Independent Test**: 在只实现“连接 + 简单心跳判定”的场景下，启动多个客户端模拟同一应用，
以及模拟长时间不发送任何消息的连接，验证系统是否按照设定的上限和超时规则关闭多余或僵尸连接。

**Acceptance Scenarios**:

1. **Given** 配置了“每个应用最大并发连接数”的限制， **When** 同一应用发起超过此限制数量的连接，
   **Then** 系统会拒绝或主动关闭多余连接，并在日志中记录原因。
2. **Given** 一个连接在一定时间内未发送任何消息且未通过心跳检查， **When** 超过约定的空闲时长，
   **Then** 系统会主动关闭该连接，并更新连接记录为“因空闲关闭”。

### Edge Cases

- 当客户端在连接建立后立即断开时，系统如何保证连接记录不会长期保留为“活动”状态？
- 当客户端在认证前就发送业务消息时，系统如何处理这类消息（例如丢弃 / 缓存 / 返回错误）？
- 当同一应用在短时间内频繁建立和关闭连接时，系统是否需要节流或暂时阻止新的连接请求？

## Requirements *(mandatory)*

### Functional Requirements

- **FR-001**: 系统必须在客户端成功建立 WebSocket 连接时创建一条新的“客户端连接记录”，
  并为该连接分配一个可用于追踪的唯一连接标识（client_id）。
- **FR-002**: 系统必须为每条连接记录维护清晰的生命周期状态（例如：连接中、认证中、可用、已关闭），
  并在连接关闭时更新状态，同时记录关闭原因和关闭时间。
- **FR-003**: 系统必须只在认证成功之后，才允许将应用标识、用户身份、租户等认证上下文绑定到连接记录上。
- **FR-004**: 系统必须阻止未完成认证的连接将业务消息当作“可用状态”消息处理，并通过错误返回或关闭连接
  的方式进行反馈。
- **FR-005**: 系统必须提供按业务实体限制并发连接数的能力；在本次特性中，业务实体指应用标识（rt_app），
  超过限制的连接必须按照策略被拒绝或被立即关闭，并附带适当的关闭原因。
- **FR-006**: 系统必须能够检测在超过可配置阈值后仍保持空闲（无业务消息、无有效心跳）的连接，
  并对其进行优雅关闭，同时在连接记录中标记为“空闲超时关闭”。
- **FR-007**: 系统必须对连接生命周期中的关键事件（创建、认证成功 / 失败、关闭、策略性拒绝）进行日志记录，
  以便追踪单个连接的行为；日志中不得暴露敏感凭证（如原始 token）。

### Key Entities *(include if feature involves data)*

- **Client Connection Record**: 抽象表示一个客户端 WebSocket 连接的完整生命周期，包含连接 ID、
  当前状态、创建时间、关闭时间、关闭原因，以及与之绑定的应用标识、用户身份和租户等上下文信息。
- **Business Entity Identifier**: 用于限制并发连接与进行资源隔离的业务维度。在本次特性的初始版本中，
  按应用标识（`rt_app`）作为并发限制的主维度，后续可以在此基础上扩展到用户级别或租户级别。

## Success Criteria *(mandatory)*

### Measurable Outcomes

- **SC-001**: 在典型负载下，所有成功建立的客户端连接都能在连接建立后立即获得唯一连接标识，
  且连接关闭后，对应记录会在合理时间内（例如 1 分钟内）被更新为“已关闭”。
- **SC-002**: 在配置了“每个应用最大并发连接数”的前提下，超过限制的连接请求全部被拒绝或立即关闭，
  且不会影响其他应用的连接正常建立。
- **SC-003**: 在配置了空闲超时时间的情况下，超过该空闲时间且未通过心跳检查的连接全部被关闭，
  不再占用系统的可用连接配额。
- **SC-004**: 运维人员可以通过查看日志或后续的观测手段，清楚追踪任意一个连接从建立到关闭的关键事件，
  用于问题定位和审计。
