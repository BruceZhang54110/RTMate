# Feature Specification: 客户端连接管理（建立连接阶段）

**Feature Branch**: `001-manage-client-connection`  
**Created**: 2025-12-30  
**Status**: Draft  
**Input**: 用户描述: "基于目前的进度 rtmate-server ClientConnection ，开发发起连接时对连接进行管理"

## User Scenarios & Testing *(mandatory)*

### User Story 1 - 连接建立时创建并跟踪客户端连接 (Priority: P1)

当业务方的客户端应用第一次通过 WebSocket 发起连接时，系统会为该连接创建一条"客户端连接记录"，
分配唯一连接标识，并在其整个生命周期内进行跟踪。连接池只存放已认证通过的连接。

**Why this priority**: 如果没有清晰、可靠的连接记录，就无法在后续实现认证、订阅、广播、限流等能力，
这是实时内核最基础的能力之一，必须优先实现。

**Independent Test**: 仅启动最小实时内核，接入一个测试客户端建立连接，不进行任何业务消息收发，
验证是否自动创建连接记录、分配唯一连接 ID，并在连接关闭后将状态更新为“已关闭”。

**Acceptance Scenarios**:

1. **Given** 系统处于运行状态且没有现有连接， **When** 一个客户端成功完成 WebSocket 连接建立并通过认证，
   **Then** 系统为该连接创建一条连接记录纳入连接池，并分配唯一连接 ID。
2. **Given** 一个已有连接处于连接池中（已认证）， **When** 客户端主动关闭连接或网络异常断开，
   **Then** 系统将对应连接记录从连接池移除，并记录关闭原因和关闭时间。

---

### User Story 2 - 基于连接记录维护认证与上下文 (Priority: P2)

当客户端建立连接并完成认证后，系统会在连接记录中绑定必要的身份 / 应用 / 租户等上下文信息，
以便后续所有消息处理都能基于这一上下文进行授权与隔离。

**Why this priority**: RTMate 面向多应用 / 多租户场景，连接如果没有绑定身份与上下文，
后续的频道、Presence、广播都会面临越权和混淆风险。

**Independent Test**: 在只实现“连接 + 简单认证”的场景下，通过一个测试客户端完成认证，
验证连接记录中是否包含应用标识、用户身份以及其他上下文字段；在认证失败时不写入这些信息。

**Acceptance Scenarios**:

1. **Given** 一个新建连接已建立 WebSocket 连接， **When** 客户端发送合法的认证消息并通过验证，
   **Then** 系统在连接池中创建 ClientConnection 记录，绑定应用标识 rt_app，并记录认证时间。
2. **Given** 一个新建连接已建立 WebSocket 连接， **When** 客户端发送无效或过期的认证信息，
   **Then** 系统不会将连接加入连接池，直接关闭连接并记录失败原因。

---

### User Story 3 - 按租户维度的并发连接限制 (Priority: P2)

当同一租户（应用）发起过多连接时，系统能够根据配置的限制进行控制，例如限制每个租户的最大并发连接数，
超过限制的新连接请求被拒绝，以保护内核资源并避免影响其他租户。

**Why this priority**: 在多租户的实时服务中，不受控的连接增长会迅速消耗资源，
导致性能问题甚至影响整体服务稳定性，需要基于连接管理能力做基础保护。

**Independent Test**: 启动多个客户端模拟同一租户的多个连接，配置最大并发连接数限制，
验证系统是否按照设定的上限拒绝超额连接。

**Acceptance Scenarios**:

1. **Given** 配置了"每个租户最大并发连接数"的限制， **When** 同一租户发起超过此限制数量的连接，
   **Then** 系统会拒绝新的认证请求，并在日志中记录原因。
2. **Given** 租户 A 的连接已达到上限， **When** 租户 B 发起新的连接，
   **Then** 系统允许租户 B 的连接成功建立，不受租户 A 的限制影响。

### Edge Cases

- 当客户端在连接建立后立即断开时，系统如何保证连接记录不会长期保留为“活动”状态？
- 当客户端在认证前就发送业务消息时，系统如何处理这类消息（例如丢弃 / 缓存 / 返回错误）？
- 当同一应用在短时间内频繁建立和关闭连接时，系统是否需要节流或暂时阻止新的连接请求？

## Requirements *(mandatory)*

### Functional Requirements

- **FR-001**: 系统必须在客户端成功建立 WebSocket 连接时创建一条新的“客户端连接记录”，
  并为该连接分配一个可用于追踪的唯一连接标识（client_id）。
- **FR-002**: 系统必须为每条连接记录维护标识和关联信息，在认证成功后将连接纳入连接池进行管理。
  连接关闭时清理对应记录，同时记录关闭原因和关闭时间。
- **FR-003**: 系统必须只在认证成功之后，才允许将应用标识、用户身份、租户等认证上下文绑定到连接记录上。
- **FR-004**: 系统必须阻止未完成认证的连接将业务消息当作“可用状态”消息处理，并通过错误返回或关闭连接
  的方式进行反馈。
- **FR-005**: 系统必须提供按租户标识（`rt_app`）限制并发连接数的能力；
  超过限制的认证请求必须被拒绝，并在日志中记录原因。
- **FR-006**: 系统必须对连接生命周期中的关键事件（创建、认证成功 / 失败、关闭、策略性拒绝）进行日志记录，
  以便追踪单个连接的行为；日志中不得暗露敏感凭证（如原始 token）。

### Key Entities *(include if feature involves data)*

- **Client Connection Record**: 抽象表示一个客户端 WebSocket 连接的完整生命周期，包含连接 ID、
  当前状态、创建时间、关闭时间、关闭原因，以及与之绑定的应用标识、用户身份和租户等上下文信息。
- **Business Entity Identifier**: 用于限制并发连接与进行资源隔离的业务维度。在本次特性的初始版本中，
  按应用标识（`rt_app`）作为并发限制的主维度，后续可以在此基础上扩展到用户级别或租户级别。

## Success Criteria *(mandatory)*

### Measurable Outcomes

- **SC-001**: 在典型负载下，所有成功完成认证的客户端连接都能纳入连接池并获得唯一连接标识，
  且连接关闭后，对应记录会在合理时间内（例如 1 分钟内）被从连接池移除。
- **SC-002**: 在配置了"每个租户最大并发连接数"的前提下，超过限制的新认证请求全部被拒绝，
  且不会影响其他租户的连接正常建立。
- **SC-003**: 运维人员可以通过查看日志或后续的观测手段，清晰追踪任意一个连接从创建到关闭的关键事件，
  用于问题定位和审计。
